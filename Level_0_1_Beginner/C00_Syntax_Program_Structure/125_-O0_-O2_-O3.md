# -O0, -O2, -O3 optimization flags

> **Level:** 0 — Absolute Beginner  
> **Category:** C00 — C++ Syntax & Program Structure  
> **Topic:** syntax

---

## Problem Statement
Understand compiler optimization levels and when to use each.

## Optimization Levels
```
Flag    Name           Speed     Debug    Binary Size   Use Case
----    ----           -----     -----    -----------   --------
-O0     No optimize    Slowest   Best     Largest       Debugging
-O1     Basic          Faster    Good     Smaller       Quick builds
-O2     Standard       Fast      Poor     Smaller       Release builds
-O3     Aggressive     Fastest   Poor     May be larger Max performance
-Os     Size optimize  Fast      Poor     Smallest      Embedded/mobile
-Og     Debug optimize Faster    Good     Medium        Debug + some speed
```

## What Optimizations Do
```cpp
#include <iostream>

int main() {
    // With -O0: compiler does exactly what you wrote
    // With -O2: compiler may:
    //   - Eliminate dead code
    //   - Inline functions
    //   - Unroll loops
    //   - Reorder instructions
    //   - Use CPU-specific instructions

    int x = 5;
    int y = x * 2;      // May be computed at compile time → y = 10
    int z = y + 0;      // May be eliminated → z = y

    for (int i = 0; i < 4; ++i) {   // May be unrolled to 4 copies
        std::cout << i << "\n";
    }

    return 0;
}
```

## Debugging Impact
```bash
# Debug build — variables visible in debugger
g++ -g -O0 main.cpp -o debug_build

# Release build — debugger shows "<optimized out>"
g++ -g -O2 main.cpp -o release_build

# Compromise — some optimization, still debuggable
g++ -g -Og main.cpp -o balanced_build
```

## Performance Example
```bash
# Compile same program at different levels
g++ -O0 heavy_math.cpp -o prog_O0
g++ -O2 heavy_math.cpp -o prog_O2
g++ -O3 heavy_math.cpp -o prog_O3

# Benchmark
time ./prog_O0    # Slowest
time ./prog_O2    # Much faster (often 2-10x)
time ./prog_O3    # Slightly faster than O2 (sometimes)
```

## Key Takeaways
1. `-O0` for debugging — no optimization, full debugger support
2. `-O2` for release — best balance of speed and reliability
3. `-O3` for max performance — may increase binary size
4. `-Og` for debug+speed — compromise option
5. Never debug with `-O2`/`-O3` — variables get optimized away

## Common Mistakes
- Debugging with `-O2` → "variable optimized out" in gdb
- Expecting `-O3` to always be faster than `-O2` → not guaranteed
- Not optimizing release builds → shipping slow code
- Relying on undefined behavior that "works" at `-O0` but breaks at `-O2`
